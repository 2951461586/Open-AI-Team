# Report Retention Policy

Status: active
Scope: decide which reports belong in the public repo story and which should remain generated/maintainer-local.

---

## 1. Rule of thumb

`reports/` should **not** be a long-lived dumping ground in the open-source repo.

Public-facing, durable conclusions should live under `docs/oss/` or `docs/archive/`.
One-off scans, local audits, and execution scratch reports should remain generated/local and stay out of the main repo story.

---

## 2. Keep vs move vs ignore

### Keep as durable docs
Move or rewrite into `docs/oss/` when a report contains:
- repo authority decisions
- open-source release framing
- contributor-relevant cleanup guidance
- durable retirement policy
- long-term migration state worth linking from docs

Examples from current local reports that are better expressed as docs:
- open-source remediation progress
- directory retirement / cleanup strategy
- high-noise directory disposition

### Archive if historically useful
Move to `docs/archive/` when a report is:
- historically informative
- tied to a specific migration phase
- no longer part of the active first-read path

Examples:
- one-time cleanup batch notes
- early retirement inventories once superseded

### Ignore as generated/local
Keep out of git when a report is:
- machine-generated
- local-environment-specific
- easy to regenerate
- primarily a maintainer sanity artifact

Examples:
- hygiene audit snapshots
- local run inventories
- temporary cleanup execution reports
- runtime/path dumps

---

## 3. Current repo recommendation

### Recommended durable docs under `docs/oss/`
- `next-step-execution-pack.md`
- `open-source-release-engineering.md`
- `repo-authority.md`
- `what-is-ai-team-runtime.md`
- `deerflow-openhanako-comparison.md`
- this file: `report-retention-policy.md`

### Recommended archive candidates
Current untracked `reports/*.md` files should not be committed as-is.
If any content is worth preserving, rewrite the distilled conclusion into:
- `docs/oss/` for active guidance, or
- `docs/archive/` for historical reference.

### Recommended ignore candidates
Treat the whole local `reports/` directory as generated by default.

---

## 4. Contributor-facing principle

External contributors should not need to read `reports/` to understand:
- where authority lives
- how to run the project
- how to contribute
- what is optional vs primary

If a report is important enough for contributors, it should become documentation.

---

## 5. Bottom line

For this repository:

> `docs/oss/` is for durable open-source guidance.
> `docs/archive/` is for historical material.
> `reports/` is generated/local unless explicitly promoted.

## AI Team Runtime 第二阶段推进
- 已新增第一阶段回归脚本：`orchestrator/scripts/team-runtime-phase1-regression.sh`，并已实跑通过，可作为当前阶段最小回归基线。
- 已新增配套 playbook：`_ops/ai-team-runtime-phase1-playbook-2026-03-08.md`，固化了回归步骤、常见异常判断与最小回滚方式。
- 第二阶段已从“只有 team runtime 账本与状态推进”推进到“具备最小角色协作消息链”。
- 当前已依次补齐：
  - `task.assign -> planner`
  - `review.request -> critic`
  - `decision.request -> judge`
  - `decision.final`
- 当前 `team_task` 自动链已可在 runtime 内统一完成：
  - planner 自动生成初始 plan
  - critic 自动生成 review
  - judge 自动生成 decision
  - task 自动推进到 `done / revision_requested`
- 实测已跑通 runtime 内统一自动闭环：复杂任务进入 `team_task` 后，可自动完成 `task.assign -> plan.submit -> review.request -> review.result -> decision.request -> decision.final -> done`。
- 当前系统最准确状态：已从“编排器 + 规则约束”升级为“中心治理型、runtime 内自动闭环的最小 AI 团队系统雏形”。
- 但当前 planner / critic / judge 仍主要是 runtime 内规则生成，不是独立 agent session 真执行。

## 下一阶段明确方向
- 下一阶段核心跃迁点不是继续堆规则，而是让 `planner / critic / judge` 从 runtime 内规则角色升级为真实 agent 执行体。
- 当前明确建议：
  - 需要新增 `planner agent` 这个角色实体。
  - 但**不建议立刻拆独立物理工作空间**；应先做 agent identity / session isolation，再视需要补 memory scope isolation，最后再考虑 workspace isolation。
- 推荐路线：
  - Phase 2.1：新增 planner agent 执行通道，让 `task.assign -> planner session` 真驱动产出。
  - Phase 2.2：给 planner 单独 memory scope。
  - Phase 2.3：视并发写文件和权限边界需要，再拆独立 workspace。
- 一条重要环境经验已确认：当前宿主机脚本应优先使用 `python3`，不要默认使用 `python`。

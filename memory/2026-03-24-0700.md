## 源码链路
我把源码线也串起来了：

### 1) doctor 的 gateway auth 检查
它走的是这条本地链：

- `evaluateGatewayAuthSurfaceStates(...)`
- `collectGatewayAssignments(...)`
- `resolveSecretInputWithEnvFallback(...)`

这里的关键点是：

- `evaluateGatewayAuthSurfaceStates(...)` 只是在判断 **surface 是否 active**
- 真正到值解析时，走的是 **本地 `resolveSecretInputWithEnvFallback(...)`**
- 这条函数本地解析失败后，**只会再看 env fallback**
  - 对 gateway token 来说就是：
    - `OPENCLAW_GATEWAY_TOKEN`

### 2) 它**没有**走 active gateway 的 runtime snapshot / `secrets.resolve`
而我们前面确认过，别的 command-secret 流程是有 `resolveCommandSecretRefsViaGateway(...) -> callGateway("secrets.resolve")` 这条能力的。

但 **doctor 这条 gateway auth warning 不是那条链**。  
它更像是：

> **本地 config/CLI 视角的 SecretRef 可用性检查**  
> 不是 “活着的 gateway 当前其实能不能正常服务” 的检查。

### 3) 这也解释了为什么现象看起来矛盾
会出现这种组合：

- daemon 明明活着
- token 明明在 secrets.json 里
- rpc 明明通
- 但 doctor 还说 unavailable

因为它们看的根本不是同一条路径。

## 规避方案
我给你按“风险/侵入性”排一下：

### 方案 A：**最推荐，最干净**
只在需要跑 doctor 时，**临时注入 env**，不做永久明文复制。

例如：

```bash
OPENCLAW_GATEWAY_TOKEN="$(
python3 - <<'PY'
import json
from pathlib import Path
obj = json.loads(Path('/root/.openclaw/secrets.json').read_text())
print((((obj.get('providers') or {}).get('gateway') or {}).get('token')) or '')
PY
)" openclaw doctor --non-interactive
```

优点：
- 不改现有 SecretRef 设计
- 不把 token 永久写进 `openclaw.json`
- 不需要改 daemon 工作方式
- 能稳定压掉这条误报

---

### 方案 B：给 CLI 常驻 env
把 `OPENCLAW_GATEWAY_TOKEN` 在 shell 环境里长期导出。

优点：
- 以后手跑 doctor 不用再包一层

缺点：
- token 变成长期 shell env，暴露面更大
- 我个人不太推荐，除非你就是想换取省事

---

### 方案 C：本地 patch / 上游修复
真正从代码层修，有两个方向：

#### C1. doctor 改成优先走 active gateway 的 `secrets.resolve`
也就是对 `gateway.auth.token` 这类 surface，不只看本地 CLI 解析，还允许参考 active gateway runtime snapshot。

这是**语义最一致**的修法。

#### C2. doctor 对 gateway auth surface 单独放宽
如果：
- `gateway status --json` 正常
- gateway auth 是 SecretRef
- 本地 secret provider 配置完整

那就别把它报成 unavailable，最多降成 info/warn-note。

这是**侵入更小**但没那么优雅的修法。

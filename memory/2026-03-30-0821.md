**一句话：balanced 模式 + weighted-random 选择 + 单并发**

---

### 配置项（env）
| 参数 | 值 |
|---|---|
| `ROTATION_MODE_MULERUN` | `balanced` |
| `MAX_CONCURRENT_REQUESTS_PER_KEY_MULERUN` | `1` |

### 路由流程

**① 过滤** — 排除不可用 key
- `key_cooldown_until > now` → 跳过（包括 billing_exhausted 的 6h cooldown 和手动禁用）
- `model_cooldowns[model] > now` → 跳过
- fair_cycle 中已标记 exhausted 的 → 跳过

**② 分层** — 可用 key 按并发状态分两层
- **Tier 1**：完全空闲（当前无任何模型在用）— 优先
- **Tier 2**：当前有请求但未达并发上限（本机上限=1，所以 Tier 2 实际不会被选中）

**③ 选择** — balanced + tolerance > 0 → `weighted-random`
```
weight = (max_usage - 该key_usage) + tolerance + 1
```
- `tolerance = 3.0` → 使用少的 key 权重更高，但不绝对
- 用 `random.choices(weights=...)` 加权随机抽一个
- 效果：**负载大致均衡，但不是严格轮询**，有一定随机性防指纹

**④ 失败处理**
- `billing_exhausted`（402）→ key-level cooldown 6h，立即轮换到下一个
- `auth_error`（401）→ cooldown 5min
- 服务端错误（500/503）→ 指数退避重试（同 key 最多 2 次），超出则轮换
- 所有 key 耗尽 → 返回聚合错误

### 为什么是这个策略
- `balanced` 适合同一账号下多 key 等权重场景
- `tolerance=3.0` 保证即使某个 key 多用了两三次，其它 key 仍有概率被选中
- 单并发（`MAX_CONCURRENT=1`）确保一个 key 同时只处理一个请求，避免 MuleRun 侧 rate limit

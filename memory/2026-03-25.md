- P2（最小读 RPC 对照矩阵）source-first 结论：`openclaw status --json` 可返回，但其 gateway 段明确显示 `reachable=false` 且 `error="missing scope: operator.read"`；而 `openclaw health --json` / `openclaw system presence --json` / `openclaw nodes status --connected` 都失败（RC=1，分别表现为正常关闭/timeout）。
- 直连 `GatewayClient` 后已证实：不是统一 timeout。握手 `hello-ok` 正常、`health` 方法可成功返回；但 `status`、`system-presence`、`config.get`、`node.list` 会立即返回 `GatewayClientRequestError: missing scope: operator.read`，`node.pair.list` 立即返回 `missing scope: operator.pairing`。
- 即使客户端显式请求全 operator scopes（`operator.admin/read/write/approvals/pairing`），上述方法仍被服务端拒绝，因此当前更准确归因是：**gateway 侧授予/识别到的 scope 集异常或方法授权映射异常**，而不是 CLI token 来源冲突，也不是 handler 普遍超时。
- P3（服务端授权映射 + granted scopes 追踪）source-first 结论：服务端 `authorizeGatewayMethod()` 对 `health` 明确写了特判 `if (method === "health") return null;`，因此 `health` 能过不是因为它拿到了 `operator.read`，而是因为它被授权检查豁免。
- 同一段服务端源码还明确显示：除 `health` 外，`status` / `system-presence` / `config.get` / `node.list` 都被归类到 `READ_SCOPE`，`node.pair.list` 被归类到 `PAIRING_SCOPE`；且 `authorizeOperatorScopesForMethod()` 会在 scopes 中缺对应项时直接返回 `missing scope`。
- 握手根因链已钉实：在 `evaluateMissingDeviceIdentity()` 路径上，若客户端**没有 device identity**，且是普通 operator（非 Control UI 特判），虽然 `roleCanSkipDeviceIdentity(role, sharedAuthOk)` 会允许其连接继续，但紧接着 `clearUnboundScopes()` 会把请求的 scopes 清空为 `[]`。后续 `authorizeGatewayMethod()` 读取的正是这个被清空后的 `client.connect.scopes`。
- 这就解释了现象：当前 CLI / 直连 GatewayClient 连接能成功（shared token auth OK，允许 operator 跳过 device identity），但除了被豁免的 `health` 外，其余需要 `operator.read` / `operator.pairing` 的方法都会立即报 `missing scope`。
- P4（device identity 调用链纠偏）source-first 结论：这不是“新机没有 identity 文件”。本机实际存在 `/root/.openclaw/identity/device.json`、`/root/.openclaw/identity/device-auth.json`、`/root/.openclaw/devices/paired.json`；其中 `device-auth.json` 已有当前 deviceId 的 `operator` token，scope 完整包含 `operator.admin/read/write/approvals/pairing`；`paired.json` 中该 device 也已绑定 `operator` 角色与全套 scopes。
- 需要纠正的点：虽然 `shouldAttachDeviceIdentityForGatewayCall()` 对 `127.0.0.1` / `::1` / `localhost` + token/password 返回 false，导致 `callGateway()` 传入 `deviceIdentity: void 0`，但 `GatewayClient` 构造器会把 `undefined` 自动补成 `loadOrCreateDeviceIdentity()`；因此**不能据此推断所有 loopback+token 调用最终都没带 device**。
- 当前被源码明确钉实的，是 `probe/status` 这条探测链（`daemon-cli.js` / `probe-auth-*.js` 中的 `probeGateway()`）：它在 loopback URL 下会计算 `disableDeviceIdentity=true`，并显式传入 `deviceIdentity: null`。而 `GatewayClient` 构造器对 `null` 的处理是“不要自动加载 identity”。
- 因此已证实的结论应改写为：**`openclaw gateway status` / `openclaw gateway probe`（以及复用这条 probe 链的状态探测）在本机 loopback 上会主动禁用 device identity，进而非常容易触发 `missing scope: operator.read` 的假阴性。** 这条证据足够支撑修复矩阵中的“非 loopback URL 绕法”和“最小补丁法”；但对所有一般 `callGateway` 读 RPC 是否同因，还需要单独继续回源，不应混为一谈。
